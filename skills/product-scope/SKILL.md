# Product Scope

先把模糊需求变成可执行、可验收、不过度设计的开发任务。不要直接写代码。

## 触发场景

- "先分析需求" / "做功能拆解" / "不要过度设计"
- "批判性产品思考" / "收敛范围" / "给出可行方案"
- 需求影响产品体验、权限、数据、API 或架构
- 用户描述的是目标，而不是明确实现步骤
- 存在多个可能方案
- 需求看起来可能过大、过抽象或不符合当前项目能力

不适用于：简单修复、明确一行配置改动、用户明确只要机械性变更。

## 工作原则

- 先问"要解决什么问题"，再问"要实现什么功能"
- 区分事实、假设、经验判断和待验证信息
- 不追求完整愿景，优先收敛到最小可交付版本
- 明确不做什么，防止需求膨胀
- 优先从仓库证据出发：现有代码、测试、文档、配置、接口、日志
- Agent 直接读取代码库证据，不询问"请帮我看一下..."

## 必须产出

```md
# Product Scope Brief

## 1. 问题定义
用户真正要解决的问题是什么？

## 2. 目标用户 / 受影响方
谁使用？谁维护？谁会受影响？

## 3. 事实依据
来自代码、文档、测试、配置的直接证据（引用文件路径和行号）。

## 4. 假设与验证方式
| 假设 | 判断依据 | 如何验证 |
|---|---|---|

## 5. 非目标
本次明确不做什么。

## 6. MVP 范围
本次只交付什么最小结果。

## 7. 备选方案
| 方案 | 优点 | 缺点 | 适用性 |
|---|---|---|---|

## 8. 推荐方案
推荐哪一个？为什么不是更复杂的方案？

## 9. 验收标准
- [ ] 可以被 Agent 自动测试或人工验证的标准 1
- [ ] 可以被 Agent 自动测试或人工验证的标准 2

## 10. 风险与回滚
主要风险是什么？如果方向错了，如何撤销？
```

## 批判性检查

在给出推荐方案前，必须自问：

```
这个功能是否真的必要？
有没有更小替代方案？
是否可以复用已有模块？
是否引入了不必要依赖？
是否制造了长期维护负担？
哪些部分是经验判断而不是代码证据？
如果只能用 20% 工作量达成 80% 效果，方案是什么？
```

## 停止条件

已得到以下全部内容时停止，等待进入 implementation-loop：

- 明确目标
- 明确非目标
- 明确 MVP
- 明确验收标准（可被 Agent 自动验证）
- 明确主要风险和回滚方式

不要继续扩展成完整产品规划，不要输出长期路线图，除非用户明确要求。
