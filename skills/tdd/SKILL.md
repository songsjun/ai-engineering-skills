# TDD（测试驱动开发）

先写失败的测试，再写代码让它通过，再清理。

**Agentic 原则：** Agent 直接运行测试，观察 RED/GREEN 状态，完成完整循环。不等待用户反馈测试结果。Agent 持续推进直到验收标准全部通过，或遇到需要架构决策的分叉点时才停止并报告。

## 核心循环

```
Agent 写测试 → Agent 运行（期望 RED）→ Agent 写最小实现 → Agent 运行（期望 GREEN）→ Agent 重构 → Agent 运行（期望 GREEN）→ 下一轮
```

TDD 的设计价值来自真实的 RED→GREEN 过渡，Agent 必须**实际运行**测试，不能跳过。

## 开始前

确认（直接读取 package.json / pyproject.toml 等确认，不询问用户）：
1. 使用什么测试框架？
2. 运行测试的命令是什么？
3. 需要实现什么行为？（如不明确，问用户一句话描述）
4. 输入和期望输出是什么？
5. 有哪些边界情况？（null/空值、边界值、错误条件）

## RED — 写一个失败的测试

生成描述预期行为的最简单测试：
- 每个测试只测一件事
- 命名：`[场景] should [期望结果]` 或 `[方法名]_[条件]_[期望]`
- 使用有意义的真实值（不要 `testValue1`，要 `"user@example.com"`）

Agent 立即运行测试，确认：
- 因缺少 import/class 而失败：正常，生成存根，继续运行
- 因没有实现而失败：正确的 RED 状态，进入 GREEN
- 没有实现就通过：测试逻辑有问题，重新审视测试

**不在确认 RED 之前写实现代码。**

## GREEN — 写让测试通过的最少代码

生成最简单的能让测试通过的实现：
- 不写超出测试要求的代码
- 不优化
- 不处理还未测试的情况

Agent 运行**所有测试**（不只是新写的），确认：
- 新测试通过
- 现有测试没有回归
- 如有回归，先修复，再继续

## REFACTOR — 清理

在测试全部通过的保护下清理：
- 消除重复
- 改善命名
- 简化逻辑

Agent 运行全套测试确认没有回归。**不在 REFACTOR 引入新行为。**

## 报告节奏

Agent 完成以下任一条件时，向用户报告进度，不是每个循环都报告：
- 一个完整功能（多个循环）完成
- 遇到接口决策（不同选择会导致显著不同的实现方向）
- 遇到无法自动验证的边界情况
- 测试无法通过且原因不明

```md
## TDD 进度报告

### 已完成循环
| 测试 | RED 确认 | GREEN 确认 | 重构 |
|---|---|---|---|
| [测试名] | ✓ | ✓ | ✓ |

### 当前覆盖
- 主路径：[已测 / 未测]
- 边界情况：[已测 / 未测]

### 阻塞项（如有）
[需要人工决策的具体问题]

### 下一轮计划
[下一个要覆盖的行为]
```

## 测试分层规则

默认写单元测试。在模块边界（DB、API、外部服务）添加集成测试。E2E 只覆盖失败后果最严重的核心路径，数量尽可能少。

## 边界情况——必须测试

对任何非平凡函数：
- 空输入 / null / undefined
- 边界值（0、最大值、最小值）
- 无效输入（动态类型语言）
- 错误/异常路径

## 遗留代码

TDD 不直接适用于没有测试的已有代码。推荐做法：

1. **理解阶段：** 读代码，理解它实际做什么（不是文档说的）
2. **特征测试：** 为现有行为写测试——测试**现在的实际行为**，即使行为有问题。防止修改时意外回归。
3. **安全修改：** 在测试保护下修改
4. **逐步改进：** 有测试覆盖后，再逐步向 TDD 靠拢

## 多 Subagent 并行

对有多个**真正独立**单元的复杂功能，可并行生成：

```
给每个 subagent 的 prompt：

请为以下单元执行完整的 TDD 循环（写测试、确认 RED、写实现、确认 GREEN、重构）。

单元描述：[具体说明这个函数/类的职责和接口]
测试框架：[如 Jest / pytest / cargo test]
语言/环境：[如 TypeScript / Python 3.11]
相关类型/接口定义：[粘贴相关类型]
边界情况必须覆盖：[列出]

请输出：
1. 测试代码
2. 实现代码
3. 运行结果（RED 和 GREEN 的实际输出）
4. 遗漏的边界情况说明

前提：只有单元之间真正不依赖彼此的接口时才并行。
```

**并行完成后，主 Agent 必须执行集成步骤：**
1. 合并所有 subagent 输出到项目代码库
2. 处理命名冲突和接口不兼容（如有）
3. 在项目根目录运行**完整测试套件**（不只是各单元的隔离测试）
4. 全部通过后进入 review-gate

## 常见错误

| 错误 | 后果 |
|------|------|
| 实现之后再写测试 | 不是 TDD，失去测试驱动设计价值 |
| 测试实现细节而非行为 | 重构时测试频繁失败，成为负担 |
| 跳过确认 RED 直接写实现 | 无法证明测试有效检测了问题 |
| 跳过 REFACTOR | 技术债积累，循环越来越慢 |
| 过度依赖 mock | 集成问题漏测，产生虚假安全感 |
